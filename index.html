FUNDAMENTALS

System Design Building Blocks

Master these 12 core components before solving any design problem. Every system is composed of these primitives.

⚖️ Load Balancer

Distributes traffic across servers. L4 (TCP) vs L7 (HTTP). Algorithms: Round Robin, Least Connections, Consistent Hashing. Examples: Nginx, AWS ALB, HAProxy.

🗄️ Database

SQL — ACID, joins, normalized. PostgreSQL, MySQL.
NoSQL — BASE, denormalized. MongoDB (document), Cassandra (wide-column), DynamoDB (key-value).

⚡ Cache

In-memory storage for hot data. Redis, Memcached. Strategies: Cache-Aside, Write-Through, Write-Behind. Eviction: LRU, LFU, TTL.

🌐 CDN

Content Delivery Network. Serves static content from edge locations closest to users. Push vs Pull CDN. CloudFront, Akamai, Cloudflare.

📨 Message Queue

Async communication between services. Decouples producers & consumers. Kafka, RabbitMQ, SQS. Enables event-driven architecture.

🔍 Search Index

Full-text search at scale. Inverted index, tokenization, relevance scoring. Elasticsearch, Apache Solr. Used in typeahead, product search.

🗂️ Blob Storage

Store large binary objects — images, videos, files. Amazon S3, Azure Blob, GCS. Paired with CDN for delivery. Immutable, content-addressed.

🔗 API Gateway

Single entry point for all client requests. Handles authentication, rate limiting, routing, SSL termination. Kong, AWS API Gateway.

📊 Sharding

Horizontal partitioning of data. Hash-based, range-based, geo-based. Consistent hashing avoids hot spots. Each shard holds a subset of data.

📋 Replication

Copies data across multiple nodes. Leader-Follower (reads scale) or Multi-Leader. Sync vs Async replication. Trade-off: consistency vs availability.

🧮 Rate Limiter

Controls request frequency. Token Bucket, Leaky Bucket, Sliding Window. Prevents abuse, protects resources. Usually in API Gateway or middleware.

🔔 Pub/Sub

Publish-Subscribe pattern. Producers publish events; subscribers receive them. Fan-out. Google Pub/Sub, SNS, Kafka Topics. Powers notifications, feeds.

Key Theorems & Concepts

CAP Theorem — A distributed system can only guarantee 2 of 3: Consistency (all nodes see the same data), Availability (every request gets a response), Partition Tolerance (system works despite network splits). In reality, P is non-negotiable → you choose CP (banking) or AP (social media).

ACID vs BASE — ACID (Atomicity, Consistency, Isolation, Durability) for SQL transactions. BASE (Basically Available, Soft state, Eventually consistent) for NoSQL at scale.

Consistent Hashing — Maps data to a hash ring. When a node is added/removed, only K/n keys need remapping (not all). Used in DynamoDB, Cassandra, CDNs.

Read-Heavy vs Write-Heavy — Read-heavy: add caching, replicas, CDN. Write-heavy: use message queues, write-behind cache, async processing, CQRS pattern.

Horizontal vs Vertical Scaling — Vertical = bigger machine (limited ceiling). Horizontal = more machines (sharding needed, complexity increases, but practically unlimited).
APPROACH

The 45-Minute Framework

A step-by-step structure for answering ANY system design question in a 45-60 minute interview.

1 Requirements & Scope (5 min)
Ask clarifying questions to narrow the scope. Identify:
Functional requirements — What must the system do? (e.g., "Users can upload and stream videos")
Non-functional requirements — Scale, latency, availability, consistency expectations
Out of scope — Explicitly state what you won't design (e.g., "Payment is out of scope")

"Design YouTube" → You: "Let me focus on video upload, transcoding, and streaming. Should I include recommendations or search?"
2 Capacity Estimation (3-5 min)
Back-of-envelope math to understand scale:
Users: DAU, peak concurrent users
Storage: Data per user × total users × retention period
Bandwidth: Read/write ratio, request rate (QPS)

Example for Twitter: 500M DAU × 2 tweets/day = 1B writes/day ≈ 12K writes/sec. Read-heavy: 100:1 read/write ratio = 1.2M reads/sec.
3 API Design (3-5 min)
Define the key API endpoints:
// Example: URL Shortener
POST /api/v1/shorten   { longUrl: "...", customAlias?: "..." }
  → { shortUrl: "tny.co/abc123", expiresAt: "..." }

GET  /:shortCode
  → 301 Redirect to original URL
Keep it RESTful. Mention authentication, pagination, and rate limiting.
4 Data Model (3-5 min)
Define entities, attributes, and relationships. Choose SQL vs NoSQL with justification:
// URL Shortener — SQL schema
urls (
  id          BIGINT PRIMARY KEY,
  short_code  VARCHAR(8) UNIQUE,
  long_url    TEXT NOT NULL,
  user_id     BIGINT REFERENCES users(id),
  created_at  TIMESTAMP,
  expires_at  TIMESTAMP,
  click_count BIGINT DEFAULT 0
)
5 High-Level Design (10 min)
Draw the architecture diagram showing major components and data flow. This is where you connect: clients → load balancer → app servers → database/cache/queue → storage.

Start simple → evolve. Begin with a monolith that works, then scale it with caching, sharding, async processing.
6 Deep Dive (15-20 min)
The interviewer will pick 1-2 areas to go deep. Be ready to discuss:
• Database choice & sharding strategy — Why DynamoDB over PostgreSQL? How do you shard?
• Caching layer — What's cached? Cache invalidation strategy?
• Failure handling — What if the queue goes down? How do you handle partial failures?
• Bottleneck analysis — Where will the system break first at 10x scale?
7 Wrap-Up (2-3 min)
Summarize trade-offs you made. Mention what you'd improve with more time: monitoring, alerting, A/B testing, gradual rollout, multi-region deployment.
💡 Pro Tip: Interviewers evaluate your thought process, not a perfect design. Communicate trade-offs explicitly: "I'm choosing Cassandra over PostgreSQL here because we need high write throughput and can tolerate eventual consistency for likes/views."
EASY

Easy System Design Problems

Start here. These have narrow scope and well-defined inputs/outputs.

1 Design a URL Shortener (TinyURL / Bitly)
AmazonGoogleMetaMicrosoftStripeUberEasy
The #1 most universal system design question. Asked at virtually every major company.

Requirements: Given a long URL, generate a unique short URL. Redirect short URL to original. Analytics tracking. URL expiration.

Key decisions:
• ID Generation: Base62 encoding of auto-increment ID or MD5 hash (first 7 chars). Base62 gives 62⁷ = 3.5 trillion unique URLs.
• Storage: Key-value store (DynamoDB) or SQL. Short code → long URL mapping.
• Read-heavy: 100:1 read/write. Cache popular URLs in Redis. 301 (permanent) vs 302 (temporary) redirect — 301 for caching, 302 for analytics.
• Scale: Partition by hash of short code. Replicate reads across regions.
URL Shortener — High-Level Architecture (animated data flow)
2 Design a Rate Limiter
GoogleAmazonStripeLinkedInEasy
Requirements: Limit API requests per user/IP. Return HTTP 429 when exceeded. Configurable rules.

Algorithms:
• Token Bucket — bucket holds N tokens, refilled at rate R. Each request consumes 1 token. Empty bucket → reject. (Most common)
• Sliding Window Log — Store timestamps of each request. Count requests in sliding window. More accurate but memory-intensive.
• Sliding Window Counter — Hybrid. Less memory than log, more accurate than fixed window.

Where to place: API Gateway (centralized), middleware, or client-side. Distributed rate limiting uses Redis with atomic INCR + TTL.
Key detail: In distributed systems, use Redis for shared counter state across multiple app servers.
3 Design Pastebin
AmazonMicrosoftEasy
Requirements: Users paste text → get a unique URL. Text accessible via URL. Expiration. Size limit (10MB).

Design: Very similar to URL Shortener + Blob Storage. Store text content in S3/blob store (not in DB). Metadata (paste ID, owner, expiry, size) in DB. Generate unique paste ID via Base62. Cache frequently accessed pastes in Redis. Add cleanup job for expired pastes.
4 Design a Key-Value Store
AppleGoogleAmazonEasy
Requirements: put(key, value), get(key). High availability. Low latency. Scalable.

Design: Consistent hashing to distribute keys. Each node stores a partition. Replication factor of 3 for durability. Write to leader + 2 replicas. Read from nearest replica.
Conflict resolution: Vector clocks or last-write-wins. Gossip protocol for failure detection.
Real-world: DynamoDB, Riak, Redis Cluster.
5 Design a Unique ID Generator
AmazonMicrosoftUberEasy
Requirements: Generate 64-bit unique, sortable IDs at high throughput. No single point of failure.

Approaches:
• UUID — 128-bit, no coordination, but not sortable and large.
• Snowflake (Twitter) — 64-bit: [1 bit unused][41 bits timestamp][5 bits datacenter][5 bits machine][12 bits sequence]. 4096 IDs/ms per machine. Time-sortable. Industry standard.
• Database auto-increment — Simple but single point of failure. Use with range allocation (each server pre-allocates a range).

Most interviews want: Snowflake-style. Explain bit allocation and why it's time-sortable.
MEDIUM

Medium System Design Problems

Multi-component systems. These test your ability to connect building blocks and reason about trade-offs.

6 Design Instagram
MetaAmazonGoogleMedium
#1 most asked at Meta. Combines media storage, feed generation, and social graph.

Core features: Upload photos/videos, follow users, news feed, stories, likes/comments, explore page.

Key decisions:
• Media storage: S3 for originals. Generate multiple resolutions. Serve via CDN.
• News Feed: Fan-out-on-write (precompute feed for each follower at write time) for normal users. Fan-out-on-read (compute at read time) for celebrity users (millions of followers). Hybrid approach.
• Database: User/post metadata in SQL. Follower graph in graph DB or adjacency list. Activity in NoSQL (Cassandra).
• Feed ranking: ML-based ranking model considering recency, engagement, relationship strength.
Instagram — Upload & News Feed Architecture
7 Design WhatsApp / Chat System
MetaAmazonMicrosoftUberMedium
Requirements: 1-to-1 messaging, group chat, online status, read receipts, media sharing. Real-time delivery.

Key decisions:
• Protocol: WebSocket for real-time bidirectional communication. HTTP for initial connection, then upgrade to WS.
• Message flow: Sender → WS Server → Message Queue → Recipient's WS Server → Recipient. If recipient offline → store in DB, deliver on reconnect.
• Storage: Messages in Cassandra (wide-column, partition by chat_id, cluster by timestamp). User metadata in SQL.
• Group chat: Fan-out to all group members via message queue. Keep group size limited (256/1024).
• Read receipts: Track per-message status (sent, delivered, read) with timestamps.
• E2E Encryption: Signal protocol. Keys exchanged via server but server can't read messages.
8 Design Twitter / X Timeline
GoogleMetaAmazonTwitterMedium
Requirements: Post tweets, follow users, home timeline, trending topics, search.

Key challenge: Timeline generation at scale. Twitter has 500M+ tweets/day.
• Fan-out-on-write: When user tweets, push to all followers' timeline caches. Fast reads but expensive writes for celebrities (fan-out to millions).
• Hybrid: Fan-out-on-write for users with <10K followers. Fan-out-on-read for celebrities. Merge at read time.
• Trending: Streaming aggregation (Apache Storm/Flink) counting hashtag frequency in sliding time windows.
• Search: Elasticsearch with inverted index on tweet text + hashtags.
9 Design a Notification System
AmazonMetaAppleMicrosoftMedium
Requirements: Push notifications (iOS/Android), SMS, email. Priority levels. Rate limiting. Preference management.

Architecture: Event producers → Kafka → Notification Service → channel-specific senders (APNS, FCM, Twilio, SendGrid).
Key details: Deduplification (don't send duplicates), retry with exponential backoff, user preference store (opt-in/out per channel), template engine for content, analytics tracking (delivered, opened, clicked).
10 Design a Web Crawler
GoogleAmazonMicrosoftMedium
Requirements: Crawl billions of web pages. Respect robots.txt. Avoid duplicate content. Politeness (rate limit per domain).

Architecture: Seed URLs → URL Frontier (priority queue) → DNS Resolver → Fetcher (multi-threaded) → Content Parser → URL Extractor → Dedup (Bloom Filter) → Content Store.
Key details: BFS for broad coverage, DFS for deep site crawling. Use consistent hashing to assign domains to specific crawler nodes. Bloom filter for URL deduplication (O(1) lookup, false positives acceptable). Store page content in blob storage, metadata in DB.
11 Design Typeahead / Autocomplete
GoogleAmazonLinkedInSpotifyMedium
Requirements: As user types, show top 5-10 suggestions in real-time (<100ms). Rank by popularity, personalization, recency.

Data structure: Trie (prefix tree) stored in memory. Each node stores top-K suggestions for that prefix. When user types "sys" → traverse to node "s→y→s" → return pre-computed top 5 results.
Updates: Don't update trie on every query. Batch: collect query logs → MapReduce aggregation hourly/daily → rebuild trie → deploy to servers. Use Zookeeper for coordination.
Architecture: Client sends prefix → Load Balancer → Typeahead Service (in-memory trie) → return cached top-K. Multi-layer caching: browser cache (for repeated prefixes), CDN, application-level Redis cache.
Scale: Shard trie by first 2 characters. "aa-az" on shard 1, "ba-bz" on shard 2. Each shard fits in memory (~GB range for billions of queries).
Personalization: Combine global popularity score with user's recent search history. Weighted merge at read time.
12 Design E-Commerce Checkout (Amazon Cart)
AmazonStripeMicrosoftMedium
Requirements: Add to cart, inventory reservation, checkout, payment processing, order confirmation. Handle concurrent buyers for limited stock.

Key challenges:
• Inventory race condition: Two users add last item simultaneously. Solution: optimistic locking with version check OR Redis DECR for atomic stock reduction. Reserve for 15 min during checkout, release if abandoned.
• Cart service: Ephemeral (Redis for logged-in, cookies for guests). Persist cart to DB on checkout initiation.
• Order flow: Cart → create order (status: PENDING) → call Payment Service → on success: update order (CONFIRMED), reduce inventory, send confirmation email via notification queue → on failure: release reservation, update order (FAILED).
• Idempotency: Payment requests must include idempotency key. Retries don't double-charge. Store payment result keyed by idempotency_key in Redis with 24h TTL.
• Database: Orders in PostgreSQL (ACID for financial data). Product catalog in MongoDB (flexible schema). Cart in Redis (fast, ephemeral).
HARD

Hard System Design Problems

Multi-service systems with complex real-time, consistency, or scale challenges. Senior/Staff level.

11 Design YouTube / Video Streaming
GoogleAmazonNetflixMetaHard
Requirements: Upload videos, transcode to multiple resolutions, stream to millions, search, recommendations.

Upload pipeline: Client → pre-signed S3 URL (direct upload) → S3 trigger → Transcoding Queue (SQS) → Transcoder (FFmpeg workers) → generates 144p to 4K → store in S3 → update metadata DB → push to CDN.
Streaming: Adaptive Bitrate Streaming (HLS/DASH). Client requests manifest file → plays segments, switching quality based on bandwidth.
Scale: CDN for popular videos (80% of views are top 20% content). Origin server for long-tail. Preload popular videos to edge locations.
Search: Elasticsearch on title, description, tags, auto-generated captions.
Recommendations: Collaborative filtering + content-based. Served from a pre-computed recommendation cache.
YouTube — Upload Pipeline & Streaming Architecture
12 Design Uber / Ride-Sharing
UberMetaAmazonGoogleHard
Requirements: Request ride, match driver, real-time tracking, ETA, payments, surge pricing.

Key components:
• Location Service: Drivers send GPS every 3-5 sec via WebSocket. Store in Redis with geohash. Geospatial query: "find drivers within 5km radius."
• Matching: Rider requests ride → find nearest available drivers (geohash lookup) → sort by ETA → send request to closest. If declined, try next.
• ETA: Dijkstra/A* on road graph + real-time traffic data. Pre-computed contraction hierarchies for speed.
• Real-time tracking: Driver location updates → server → pushed to rider via WebSocket.
• Surge pricing: Supply/demand ratio per geohash region. Dynamic multiplier.
• Scale: Partition by city/region. Each region has its own location service cluster.
13 Design Google Docs / Collaborative Editing
GoogleMetaHard
Requirements: Multiple users edit same document simultaneously. Real-time sync. Conflict resolution. Cursor positions. Version history.

Key challenge: Conflict resolution. Two approaches:
• OT (Operational Transformation) — Used by Google Docs. Transform conflicting operations so both can be applied. Complex but battle-tested.
• CRDT (Conflict-free Replicated Data Types) — Used by Figma. Data structure that's mathematically guaranteed to converge. Simpler logic but higher storage.

Architecture: Client sends operations (insert char at position X) → Server applies OT → broadcasts transformed ops to all other clients. WebSocket for real-time. Document stored as operation log + periodic snapshots.
14 Design Google Maps
GoogleUberHard
Requirements: Map rendering, turn-by-turn navigation, ETA, real-time traffic, search places.

Map tiles: Pre-rendered at multiple zoom levels (0-21). Stored in blob storage. Served via CDN. Client requests tiles based on viewport.
Routing: Road network as weighted graph (billions of edges). A* algorithm with contraction hierarchies for fast queries. Pre-compute popular routes.
Traffic: Aggregate GPS data from millions of phones. Segment-level speed computation. Update routing weights in real-time.
Search: Geocoding (text → lat/long), reverse geocoding (lat/long → address). Elasticsearch with geospatial filters.
15 Design a Distributed Cache (Memcached/Redis)
AmazonGoogleMicrosoftHard
Requirements: In-memory key-value store. Sub-millisecond reads. Horizontal scaling. High availability.

Architecture: Consistent hashing to distribute keys across cache nodes. Virtual nodes for uniform distribution. Replication for HA (async replication to follower).
Eviction: LRU (doubly linked list + hashmap). LFU for frequency-based.
Cache stampede prevention: Lock-based (only one thread fetches from DB, others wait), probabilistic early expiration, stale-while-revalidate.
Write policies: Write-through (write to cache + DB simultaneously), Write-behind (write to cache, async batch write to DB).
16 Design a Payment Processing System (Stripe)
StripeAmazonGoogleUberHard
Requirements: Process credit/debit card payments for merchants. Support pay-in (customer → merchant) and pay-out (merchant → bank). Handle refunds. 99.99% uptime. ~1M transactions/day.

Key principles (never violate these):
• Never use floating-point for money. Store as integers in smallest unit (cents/paisa). $12.50 = 1250.
• Idempotency is mandatory. Every API call includes an idempotency_key. If client retries due to timeout, return the same result — never double-charge.
• Payments are state machines, not records. States: CREATED → PROCESSING → AUTHORIZED → CAPTURED → SETTLED (or FAILED/REFUNDED). Enforce valid transitions only.

Architecture:
Merchant API call → API Gateway (auth, rate limit) → Payment Service → validates merchant + amount → sends to Payment Processor (Visa/Mastercard via acquirer bank) → on approval: write to Ledger DB (double-entry bookkeeping) → emit event to Kafka → Notification Service sends webhook to merchant → Settlement Service runs nightly batch to transfer funds.

Critical design details:
• Double-entry ledger: Every transaction = two entries (debit one account, credit another). Sum of all entries = 0. This is how Stripe and all financial systems ensure correctness.
• Reconciliation: Nightly job compares internal ledger with external bank statements. Flags discrepancies. Defense-in-depth against bugs and bank errors.
• Outbox pattern: Write payment record + event message in same DB transaction. Separate process reads outbox and publishes to Kafka. Guarantees at-least-once event delivery.
• PCI Compliance: Never store raw card numbers. Use tokenization — replace card with a token at entry point. Only PCI-certified systems touch raw card data.

17 Design Netflix
NetflixAmazonGoogleHard
Requirements: Stream video to 200M+ users globally. Personalized recommendations. Multi-device support. Offline downloads. Continue watching across devices.

Content Ingestion Pipeline:
Studio uploads master file → Transcoding farm (thousands of workers) → generates 2000+ versions per title (resolution × codec × bitrate × audio language) → store in S3 → replicate to Open Connect Appliances (Netflix's own CDN boxes placed inside ISPs).

Streaming Architecture:
• Client requests manifest → Steering Service selects best Open Connect server (closest, least loaded) → client streams segments via HLS/DASH.
• Adaptive Bitrate: Client continuously measures bandwidth → switches between quality levels within segments. Buffer management: keep 30-60 sec ahead.

Recommendation Engine:
Two-stage approach: (1) Candidate generation — collaborative filtering + content-based filtering produces ~1000 candidates per user. (2) Ranking model — ML model scores each candidate based on user history, time of day, device, region → return top 50-75 for each row in UI.
Pre-computed offline (Spark jobs), cached per user. Updated daily. Real-time signals (just watched X) adjust rankings via lightweight online model.

Key Netflix-specific detail: Open Connect CDN. Netflix places hardware appliances (OCAs) inside ISP networks. During off-peak hours, new content is pushed to OCAs. During peak, 95% of traffic is served from within the ISP network → near-zero latency, no internet backbone needed.

18 Design TikTok
MetaAmazonGoogleDoorDashHard
A modern question increasingly asked at FAANG (2024-2026). Combines video processing with a recommendation-first architecture.

Requirements: Upload short videos (15s-3min), infinite scroll "For You Page" (FYP), follow creators, like/comment/share, duets/stitches, creator analytics.

What makes TikTok different from YouTube: The core product IS the recommendation engine, not search or subscriptions. FYP is 100% algorithmically driven for users who follow nobody.

Upload pipeline: Video → App Server → S3 raw → Transcoding queue → generate HLS segments (multiple resolutions) → CDN. Also: extract audio (for music matching), generate thumbnails, run content moderation (NSFW/violence detection ML model) before publishing.

For You Page (FYP) Algorithm:
• Signals: Watch time (most important — did user watch full video?), replays, likes, shares, follows, comments, video info (hashtags, sounds, captions), device/location.
• Two-tower model: User embedding tower + Video embedding tower → compute similarity in real-time. Cold-start: show trending/diverse videos to new users, rapidly learn preferences from first 10 interactions.
• Serving: Pre-compute top ~500 candidates per user (batch pipeline). At request time, online ranker re-scores with latest signals → return top 30 → client pre-fetches next batch while user watches current.

Scale detail: Video metadata in Cassandra (high write throughput). Engagement events in Kafka → Flink streaming → update user embeddings in near-real-time. CDN handles 90%+ of video delivery.
19 Design a Distributed Message Queue (Kafka)
AmazonMicrosoftLinkedInUberHard
Requirements: High throughput (millions of messages/sec), durable, ordered within partition, at-least-once delivery, consumer groups, retention-based (not delete after consume).

Architecture (Kafka model):
• Topic: Logical channel for messages (e.g., "user-events", "payments").
• Partition: Topic split into N partitions. Each partition is an ordered, immutable append-only log. Partition = unit of parallelism.
• Broker: Server storing partitions. Partitions replicated across brokers (leader + ISR follower set). Leader handles all reads/writes for that partition.
• Producer: Sends message with key → hash(key) % N → routes to partition. Same key always goes to same partition (ordering guarantee).
• Consumer Group: Each partition consumed by exactly one consumer in the group. Add consumers = more parallelism (up to partition count).

Why append-only log is fast: Sequential disk writes (600MB/s) vs random writes (100KB/s). Zero-copy via sendfile() syscall. Page cache utilization. This is why Kafka beats traditional message brokers on throughput.

Delivery semantics: At-most-once (fire and forget), at-least-once (ack after process, may duplicate), exactly-once (idempotent producer + transactional consumer — Kafka supports this since 0.11).
20 Design a Search Engine (Google Search)
GoogleAmazonHard
Requirements: Index billions of web pages. Return relevant results in <200ms. Spell correction. Ranking. Snippets.

Pipeline:
1. Crawling: Web crawler fetches pages (see Problem #10). Prioritize by PageRank, freshness, link count.
2. Indexing: Parse HTML → extract text → tokenize → build inverted index (word → list of [docID, position, frequency]). Store in distributed index shards.
3. Serving: Query → tokenize → lookup each token in inverted index → intersect posting lists → score documents (TF-IDF, BM25, PageRank, freshness, user signals) → return top 10 with snippets.

PageRank: Model web as a directed graph. A page's rank = weighted sum of ranks of pages linking to it. Iterative computation until convergence. Pre-computed offline.
Scale: Shard index by document ID range. Each shard serves a subset. Scatter query to all shards → gather top results → merge and re-rank. Cache frequent queries in Redis.

📊 SQL vs NoSQL Decision Matrix
Choose SQL when: Data is relational, need ACID transactions, complex queries with JOINs, data integrity is critical (banking, e-commerce orders).

Choose NoSQL when: Massive scale, flexible schema, high write throughput, denormalized data is acceptable, simple access patterns (key-value lookups, time-series).

Specific choices: PostgreSQL (relational + JSON), Cassandra (time-series, high writes), MongoDB (documents, flexible schema), DynamoDB (managed key-value, auto-scaling), Redis (cache, sessions, leaderboards), Elasticsearch (full-text search).
⚡ Latency Numbers Every Engineer Should Know
L1 cache reference           0.5 ns
L2 cache reference             7 ns
Main memory reference        100 ns
SSD random read          150,000 ns   (150 μs)
HDD seek                10M     ns   (10 ms)
Network round trip (same DC)  500,000 ns (0.5 ms)
Network round trip (CA→NL)  150M    ns  (150 ms)
Takeaway: Memory is 1000x faster than SSD. SSD is 100x faster than HDD. Same-datacenter network is 300x faster than cross-continent. This is why caching and CDNs matter.

1 TB = 1 trillion bytes = stores ~5 billion tweets or ~5 million photos or ~20K hours of video.

🔑 Golden Rules for Interview Day
1. Never jump to solution. Spend first 5 min on requirements.
2. Start simple, then scale. Monolith → Microservices → Distributed.
3. Every decision must have a why. "I chose Cassandra because..." not "Let's use Cassandra."
4. Trade-offs > perfection. Interviewers want to hear "this gains X but costs Y."
5. Numbers matter. Estimate scale before designing. QPS, storage, bandwidth.
6. Think about failure. "What happens if this component goes down?"
7. Communicate constantly. Silence = bad. Even saying "I'm thinking about..." is better.
8. Don't memorize — understand the building blocks and compose them.
💡 Final Advice: System Design isn't about memorizing solutions — it's about composing building blocks to solve a unique problem under constraints. If you understand the 12 building blocks in Tab 1 and practice the framework in Tab 2, you can tackle ANY question they throw at you.
